## ASR Model Findings

### Olicorne SmoothQuant Parakeet TDT v3 multilingual — VERIFIED, maps to TASK-223/225
- **Source:** https://huggingface.co/Olicorne/parakeet-tdt-0.6b-v3-smoothquant-onnx (lastModified 2026-07-19; weights promoted 2026-06-28)
- **Verified facts (read from the HF card this run):**
  - **sherpa-loadable layout:** `encoder-model.int8.onnx` + `decoder_joint-model.int8.onnx` + `vocab.txt` + `nemo128.onnx` — the canonical istupakov/Parakeet-TDT layout sherpa's `OfflineRecognizer` already loads for our shipped Parakeet. Minimal wiring (point `SherpaOnnxBackend` at the dir; same architecture).
  - **Two builds:** `default` 841 MB (18 worst MatMuls kept fp32, auto-picked) / `lite` 757 MB (11 kept, opt-in).
  - **25-lang FLEURS macro WER (greedy):** default **14.27%** / lite 14.82% / istupakov stock int8 **19.40%** / fp32 12.85%.
  - **Long-audio mean WER (8 held-out ~390s speeches):** default **8.6%** / istupakov int8 **24.8%** / fp32 7.8% — SmoothQuant's biggest win is exactly where stock int8 collapses.
  - CER: default 4.47% / istupakov int8 6.87%.
  - License: **CC-BY-4.0**. Calibration: FLEURS train (all 25 langs, no labels).
- **Caveats:** (1) NOT in sherpa's model zoo — manual integration into `OfflineRecognizer` (or a re-export), same as any non-csukuangfj Parakeet. (2) Italian per-language WER is on the card's FLEURS table but **not yet extracted** — the cheap pre-check before any eval. (3) Mildly contradicts the 07-15 "soniqo" insight ("decoder-joint must stay fp32"); Olicorne ships int8 decoder-joint and claims fp32 parity with evidence — worth noting for TASK-225.
- **Score:** 7/10. **Recommendation: this IS TASK-225.** Cheapest next step: pull Olicorne's Italian per-lang WER from the card; if competitive with Distil-IT's ~4.3%, smoke-load the 841 MB default via `SherpaOnnxBackend` and run the (now-working) eval harness. The long-audio fix alone justifies the eval — our Italian voice messages are exactly where stock int8 collapses.

### GGUF-audio watch — REFINED (still blocked, clearer blockers)
- **llama.cpp now supports Gemma 4 audio** (since June: PRs #24091/#24118; issue #25889 Jul 19 shows it working with `gemma-4-E2B-it-Q4_0.gguf` + an audio mmproj). So the upstream blocker the 07-15 report cited ("llama-bro pinned old") is *partially* past — llama.cpp has it.
- **NEW hard blocker:** no public `mmproj-gemma-4-E2B-it` audio projector file exists (HF search: 0 results; `google/gemma-4-E2B-it` ships only unified safetensors; the working example used a self-converted projector).
- **llama-bro still dormant** (fork + upstream both last commit Mar 25–27).
- **Net:** the GGUF-audio unblock now needs three pieces (llama-bro submodule bump + mmproj-audio loader + a published/converted audio projector file), none shipped. Still blocked; the blocker list is clearer.

### No Italian-audio LoRA (Gilbeot technique)
No Italian-audio Gemma 4 LoRA published. Precedents remain FR/EN (Steveeeeeeen, May); two July German attempts are empty stubs. The Gilbeot→Italian technique (TASK-290) is still a green-field training project.

### Watch (not actionable)
Google's Gemma 4 **12B** uses raw-16kHz "wave projection" (no 12-conformer audio encoder) — far more efficient than E2B/E4B's conformer, but 12B is too large for mobile. If backported to E2B/E4B it would materially cut on-device audio cost. Not happening yet.
